# 对话结果图文字二次编辑设计

日期：2026-07-14

状态：已实现并通过最终聚焦验证，随 `v0.2.1` stable Release 发布，等待生产更新。本文描述在当前对话中对已生成图片进行文字描述编辑；不包含画笔、蒙版或独立专业编辑器。

## 1. 目标与范围

用户可以在当前对话的成功结果卡上点击“编辑图片”，输入新的修改描述，并按一次普通生图任务生成一张新图片。

本期必须满足：

- 只允许从当前对话中的成功结果卡发起编辑。
- 只支持文字描述编辑，不提供画笔、框选、蒙版、扩图或局部工具。
- 原图始终保留，编辑结果追加为同一对话的新一轮生成。
- 新结果显示原图缩略图和“基于此图编辑”，并且新结果可以继续编辑，不限制层数。
- 编辑任务使用提交时用户当前选择的 `system` 或 `custom` 模式。
- 编辑按一次普通生图处理，沿用现有积分、并发、预算、Key 和成功扣费规则。

明确不在本期范围内：

- 资产库、灵感库或公开图片直接发起编辑。
- 替换或删除原图。
- 免费编辑、单独编辑价格或新的计费模型。
- 多张来源图、画布、蒙版、撤销/重做和独立编辑器页面。
- 新的生成端点或第二套任务队列。

## 2. 用户交互

### 2.1 入口与编辑态

成功结果卡显示“编辑图片”操作；失败、排队中、生成中或缺少图片的卡片不显示该操作。

点击后，当前对话底部的 Composer 进入专用编辑态，不打开右侧输入抽屉。右侧“本次图片”面板保持原样，避免产生两个输入位置。

编辑态包含：

- 来源图片缩略图、来源图片 ID 和“正在编辑这张图”标识。
- 默认为空的修改描述输入框，提示用户描述想改变的内容。
- 完整的尺寸、质量和背景控件，初始值继承来源 generation 的对应参数，用户可以修改。
- 当前提交时的 Key 模式展示；不在编辑态新增第二个 Key 模式选择器。
- “取消”操作，取消后恢复普通 Composer，不创建任务。
- “生成编辑结果”操作。编辑态不重复显示积分金额文案；计费仍按现有服务端规则执行。

桌面端继续使用底部 Composer；移动端使用现有底部输入区域的可滚动布局。提交成功收到 `202` 后，编辑态关闭，页面滚动到新生成卡片。

提交前的参数错误、权限错误或网络错误不会清空输入；用户可以在编辑态中修正后重试。

### 2.2 结果与版本关系

编辑请求创建全新的 generation 和 image。原 generation、原 image 和原提示词不修改。

新结果卡包含：

- 编辑描述作为本轮提示词。
- 来源图片小缩略图。
- 文案“基于此图编辑”。
- 现有的下载、保存资产库、重试和查看原始响应操作。

重试操作复用来源图片、编辑描述和本轮参数；点击新结果卡的“编辑图片”会创建下一层来源关系。用户可以沿来源缩略图逐级查看历史版本。

## 3. 接口与数据模型

### 3.1 生成请求

继续使用 `POST /api/generate`，在现有请求契约中增加可选字段：

```ts
sourceImageId?: string;
```

约束：

- `sourceImageId` 必须是 UUID，并且与临时 `inputImageKey` 互斥。
- 客户端只能提交图片 ID，不能提交 storage key、文件路径或外部 URL。
- 编辑请求的 `prompt` 只包含用户填写的修改描述，不强制把原始提示词拼接进请求。
- `credentialMode` 使用提交时当前用户的模式，不能从来源图片推断。

### 3.2 generation 来源字段

`generations` 增加可空的 `source_image_id` 字段和查询索引。现有文生图及手动参考图任务保持 `NULL`。

该字段直接指向来源 `images.id`。由于当前 `images.generation_id` 已指向 `generations`，本期不新增循环外键；服务层在事务中执行来源存在、成功状态和用户归属校验，并通过索引保证查询可控。来源图片被删除后，历史 generation 仍保留来源 ID，但新的编辑请求必须返回不可编辑错误。

对话详情和状态契约返回可选的来源摘要：

```ts
sourceImage: {
  id: string;
  publicUrl: string;
  width: number;
  height: number;
} | null;
```

摘要只来自服务端按当前用户查询到的图片记录，不接受客户端回传的 URL。

### 3.3 worker 数据流

1. Web 在入队事务中校验 `sourceImageId`，创建带 `source_image_id` 的 generation。
2. worker 领取任务后，按 generation 的用户 ID 和来源图片 ID 读取图片记录。
3. storage adapter 从本地媒体卷或配置的兼容存储读取原图字节；浏览器不重新下载或上传原图。
4. worker 使用现有 `/images/edits` multipart 管线提交原图、编辑描述、尺寸、质量和背景。
5. 成功路径创建新的 image 记录并保留来源 generation；失败路径沿用现有错误分类和终态处理。

如果原图在入队后、worker 处理前被删除或无法读取，任务进入明确的来源不可用失败态，不产生图片，也不扣 system 积分。

## 4. 计费、状态与安全

- system 编辑请求入队时执行现有余额、并发和日预算检查；只有成功终态才执行一次 debit。
- custom 编辑请求沿用当前 custom 凭据生命周期和零积分规则；成功写入 `credits_charged_mp=0`。
- 失败、超时、来源不存在、来源读取失败和上游拒绝均不产生 system debit。
- system 单任务限制和 custom 多任务规则保持不变。
- 来源图片必须属于当前登录用户且状态为成功；越权、伪造 ID 和未成功图片统一拒绝。
- 服务端和 worker 只使用来源图片的存储记录，不把 storage key、Key 明文或上游原始响应写入普通字段、日志、事件或响应。
- UI 对来源不存在显示“这张图片已不可编辑”；其他失败继续使用现有友好错误映射。

## 5. 实现边界

前端复用 `ConversationView`、`Composer`、现有生成 hook 和对话查询缓存；新增的只是编辑态、来源摘要展示和结果卡来源标识。

后端复用 `enqueue`、generation 状态机、storage adapter、Relay `/images/edits` 和现有成功/失败事务；不新增 `/api/generate/edit`，不新增队列服务。

数据库迁移采用向后兼容的可空字段和索引。旧 generation 无来源字段时行为完全不变；部署顺序为先执行兼容迁移，再启用新 Web/worker 代码。

## 6. 验收与测试

### 前端

- 成功结果卡显示“编辑图片”，非成功卡片不显示。
- 点击后 Composer 进入编辑态，来源缩略图出现，修改描述为空，参数继承且可调整。
- 取消恢复普通 Composer；提交成功后编辑态关闭并定位到新卡片。
- 新卡片显示来源缩略图和“基于此图编辑”，新卡片可再次编辑。
- 入队或参数错误时输入和参数保留。

### 契约与服务端

- 校验 `sourceImageId` UUID、用户归属、成功状态和与 `inputImageKey` 的互斥关系。
- 验证来源图片记录写入 generation，响应和对话详情返回来源摘要。
- 验证 worker 读取服务端来源图片并调用编辑管线，不接受客户端 storage key。
- 验证来源删除/不可读、越权和伪造 ID 均不创建有效任务。

### 计费与状态

- system 成功只扣一次，失败和超时不扣。
- custom 成功不扣本站积分。
- 原图始终保留，连续编辑生成独立 image 和 generation。
- 重试复用来源图、编辑描述和本轮参数。

### 手动验收

完成一次本地流程：成功编辑、参数修改、失败重试、连续编辑和越权拒绝。使用现有测试命令加本需求新增的聚焦测试，不扩展无关回归范围。

## 7. 完成标准

当上述交互、来源关系、计费规则、安全约束和聚焦测试全部落地后，本需求视为完成。实现完成后再把本文件状态改为已实现，并在项目需求、开发和验收文档中记录实际提交与部署证据。
