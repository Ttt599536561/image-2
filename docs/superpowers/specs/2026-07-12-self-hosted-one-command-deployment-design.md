# 单机全自托管一键部署设计

状态：用户已确认采用空数据启动、服务器本地 PostgreSQL、本地图片存储、自设管理员，以及可见的 Key/密码输入。

## 目标

- Debian 服务器只依赖 Docker Engine 与 Compose，不再依赖 Neon、Supabase 或 Netlify。
- 用户运行一个安装脚本，只输入系统中转 Key、管理员邮箱和管理员密码。
- PostgreSQL、图片、备份和应用进程全部保存在同一台服务器。
- 数据库密码、认证密钥和任务加密密钥由脚本自动生成。
- Web 的 `3000` 和 PostgreSQL 的 `5432` 只存在于 Docker 网络，不占宿主机同名端口。
- 安装、恢复和升级不得静默覆盖已有数据。

不迁移旧数据库、旧用户或旧图片；首次部署从空数据开始。本期不做多机高可用、外部对象存储或自动异地备份。

## 部署体验

入口为：

```bash
sudo bash deploy/install.sh
```

脚本先检查 Debian、Docker、Compose、磁盘、端口和已有数据。检查通过后按顺序使用普通可见输入：

1. 系统中转 Key：完整回显并要求 `y/N` 确认。
2. 管理员邮箱：校验邮箱格式。
3. 管理员密码：连续输入两次，必须完全一致，长度至少 6 个字符且不超过 72 字节。
4. 显示部署摘要，再要求一次 `y/N` 确认。

用户已接受 Key 和密码可能保留在终端滚动记录、录屏或 AI 操作日志中的风险。脚本使用交互式读取，不把输入值放入 shell 命令历史，也不得启用 shell xtrace 或把交互内容写入安装日志。管理员密码仅传给初始化进程，创建账号后立即从进程环境清除，不写入长期配置。

确认后脚本自动生成内部密钥、创建权限为 `600` 的配置、启动数据库、执行迁移、创建管理员、构建并启动应用，最后执行健康检查并输出网站与 `/admin/login` 地址。

## 运行架构

Compose 包含以下服务：

- `postgres`：单机 PostgreSQL，使用私有 Docker 网络和持久化 `postgres_data` 卷，不发布宿主机端口。
- `web`：React Router SSR 与 API，容器内部监听 `3000`。
- `worker`：执行生成任务并写入数据库和媒体卷。
- `scheduler`：保持单副本，执行超时、清理、到期和对账任务。
- `caddy`：域名模式下提供 TLS、反向代理和 `/media/*` 文件服务。

`web`、`worker`、`scheduler` 共享 `media_data` 卷；Caddy 可只读挂载该卷。生产代码新增显式的 `DATABASE_DRIVER=pg` 和 `STORAGE_DRIVER=local`，不得复用 `DISPOSABLE_TEST_DB_DRIVER` 作为生产开关。

本地数据库的读与事务路径统一使用标准 `pg` 连接池，同时保留现有事务、行锁和防双花语义。内部连接地址由脚本生成，两个数据库 URL 均指向 Compose 的 `postgres:5432`。

本地媒体适配器复用现有路径穿越和符号链接防护，支持写入、读取、列表和删除。Web 始终提供安全的 `/media/*` 文件路由；内置 Caddy 启用时可直接从只读卷响应同一路径以减少应用负载。数据库中的本地图片地址使用同源相对路径 `/media/<storage-key>`，避免以后切换域名或端口导致历史图片 URL 失效，也保证现有代理模式只转发到 Web 时仍能访问图片。

## 域名与端口

域名和反向代理由部署 AI 通过脚本参数处理，不增加用户交互项：

- 域名模式：AI 传入域名；端口 `80/443` 空闲时启用内置 Caddy。
- 现有代理模式：内置 Caddy 不启动，Web 仅绑定 `127.0.0.1:<自动选择的空闲端口>:3000`，脚本输出代理上游地址。

脚本不得自动停止、重配或删除服务器上已有容器。检测到端口冲突时应停止并给出可执行的替代方案。现有服务占用宿主机 `3000` 不构成冲突，因为本项目的 `3000` 默认不发布到宿主机。

## 初始化与管理员

数据库健康后，安装脚本按固定顺序执行：

1. 对空数据库应用仓库受控迁移。
2. 使用用户输入的邮箱和密码调用现有 `seed-admin.ts`。
3. 验证业务 `users.role` 与 Better Auth `user.role` 都为 `admin`。
4. 启动 Web、worker 和单例 scheduler。
5. 验证 `/healthz` 返回 `204`，并验证管理员可通过 `/admin/login` 登录。

初始化必须幂等。管理员已存在时只验证角色，不重置密码；需要重置时使用单独的显式运维命令。

## 数据与备份

- `postgres_data` 保存数据库，`media_data` 保存生成图、上传图和灵感图。
- `deploy/backup.sh` 在宿主机权限为 `700` 的 `deploy/backups/` 中生成 PostgreSQL dump、媒体归档、校验和与版本清单。
- `deploy/restore.sh` 只向停止的应用恢复，并在覆盖前要求显式确认。
- 默认保留最近 7 份本地备份；备份目录不得位于临时容器层。
- `deploy/.env.production` 不进入普通备份包，需以权限 `600` 单独保管。

本地备份不能抵御整机或磁盘损坏。异地复制是上线后的运维增强项，不阻塞首次部署。

## 失败与升级

- 预检失败时不创建配置、卷或容器。
- 中途失败保留已创建的数据；再次运行识别阶段并安全继续。
- 检测到现有 `postgres_data`、`media_data` 或生产配置时，默认拒绝全新安装，只允许显式 `--resume`、`--upgrade` 或 `--restore`。
- 升级先备份，再构建新镜像、执行兼容迁移和滚动重启；不得自动删除旧卷。
- 健康检查失败时输出相关服务日志和恢复命令，不打印系统 Key、数据库密码或内部密钥。

## 验收

实现完成必须证明：

- 全新 Debian/Compose 环境可从空卷一键安装。
- 三项输入的顺序、可见回显、Key 确认和密码双输校验正确。
- PostgreSQL 与媒体数据在容器重建后仍存在。
- 注册、管理员登录、系统 Key 生图、图片回看和 scheduler 清理可用。
- 金额事务测试、单元测试、类型检查、密钥扫描和 Docker 构建通过。
- 本地媒体适配器覆盖路径穿越、符号链接、读写、批量删除和孤儿列表测试。
- `3000` 已被其他容器发布时，本项目仍可正常部署。
- 备份后可在空卷恢复数据库与图片，并通过健康检查。
- CI 构建最终 Compose 镜像并执行一次空数据安装 smoke。
