## 一致性规则

- “已实现”表示代码与配置已经交付；“已部署”表示生产环境已经运行；“已发布”只用于真实存在的 GitHub Release。
- 不写入服务器密钥、环境变量值、数据库内容或其他秘密。
- 命令必须继续显式使用 `docker compose --env-file deploy/.env.production`。
- 不把一次生产上线验证扩写成未实际执行的真实 Relay 付费生图或灾难恢复演练。
- 未来事项只保留真正未实现的运维增强，例如加密异地备份和多机高可用；不得把它们写成当前需求未完成。
- 同一事实只保留一个权威细节来源，其他文档使用链接或短摘要，避免多个版本漂移。

## 验证方式

完成修改后执行一次文档验证：

1. 搜索未勾选任务、`TODO`、`TBD`、旧版本号和“待实现/尚未实现”等状态词，逐项确认是否仍然成立。
2. 检查所有 Markdown 相对链接指向存在的文件。
3. 检查版本、提交、生产地址、备份 ID、更新入口和 Release 状态在各文档中一致。
4. 运行 `git diff --check`，确认没有空白错误。
5. 只提交文档文件，不混入应用代码或生产配置。

## 完成标准

仓库中任何一个常用文档入口都不会再把现有需求描述为待开发；历史规格和计划明确显示完成；部署与运维说明符合当前生产环境；GitHub Release 的未发布状态被准确保留；后续新增需求可以从新的需求变更开始，而无需继续维护本轮已完成清单。
